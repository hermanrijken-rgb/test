IA3 Weather Dashboard: Open-Meteo cURL Guide

Self-help guide for retrieving, processing and explaining live forecast data securely.

IA3 support resource

1Why this guide matters

Your IA3 prototype is a secure weather dashboard. It needs to retrieve Open-Meteo weather data, process JSON, show clear forecast information, and explain how privacy and security are considered when working with account and location data.

🎯 Learning intention

  • Understand how Open-Meteo sends weather data as JSON.
  • Use location search to find latitude and longitude.
  • Use PHP cURL to request API data more reliably.
  • Explain security choices in IA3 documentation.

✅ Success criteria

  • Test an Open-Meteo URL in a browser.
  • Identify the JSON fields needed for my dashboard.
  • Explain why cURL is better than a simple file request.
  • Recommend cURL security and reliability options.

🌦️ Important naming note

The weather API is called Open-Meteo, not Open-Mateo. The approved source URL is https://open-meteo.com/.

Use this pathway if the task feels large. Build the essential parts first, then add stronger features after the core weather data works.

3Search for locations

The forecast endpoint needs coordinates. Use Open-Meteo's geocoding API to turn a place name into latitude and longitude.

URL · Search for Cairns
https://geocoding-api.open-meteo.com/v1/search?name=Cairns&count=5&language=en&format=json

Fields to use

  • name — display name
  • latitude — needed for forecast
  • longitude — needed for forecast
  • admin1 — state or region
  • country — country
  • timezone — local time zone

⚠️ Do not assume the first result is always correct

Some places have the same name. A stronger design shows several results and lets the user choose the correct one.

JSON · Example geocoding result
{   "name": "Cairns",   "latitude": -16.923,   "longitude": 145.7663,   "country": "Australia",   "admin1": "Queensland",   "timezone": "Australia/Brisbane" }

4Use PHP cURL to request the API

For this assessment, cURL is recommended because it gives your PHP program a clear way to request JSON data from an API and then display the results in your own page.

Student note

The examples below are deliberately plain. They do not include CSS and they do not show all security or reliability choices. Use them to understand the API request process, then make your own design and explain your own technical decisions in your IA3 documentation.

Starter example 1: Display current Cairns weather

This file uses fixed Cairns coordinates. It is useful for testing whether cURL, JSON decoding and basic output are working before adding user input.

PHP · cairns_weather_basic.php
<?php $url = "https://api.open-meteo.com/v1/forecast?latitude=-16.92&longitude=145.77&current=temperature_2m,relative_humidity_2m,precipitation,wind_speed_10m,weather_code&timezone=Australia%2FBrisbane"; $curl = curl_init(); curl_setopt($curl, CURLOPT_URL, $url); curl_setopt($curl, CURLOPT_RETURNTRANSFER, true); $response = curl_exec($curl); curl_close($curl); $data = json_decode($response, true); ?> <!DOCTYPE html> <html> <head>     <title>Cairns Weather</title> </head> <body>     <h1>Cairns Current Weather</h1>     <?php if (isset($data["current"])): ?>         <p>Temperature: <?php echo $data["current"]["temperature_2m"]; ?> °C</p>         <p>Humidity: <?php echo $data["current"]["relative_humidity_2m"]; ?>%</p>         <p>Rainfall: <?php echo $data["current"]["precipitation"]; ?> mm</p>         <p>Wind speed: <?php echo $data["current"]["wind_speed_10m"]; ?> km/h</p>         <p>Weather code: <?php echo $data["current"]["weather_code"]; ?></p>     <?php else: ?>         <p>Weather data could not be loaded.</p>     <?php endif; ?> </body> </html>

How the webpage is built after the cURL request

The first PHP block requests and decodes the API data. After that, the file changes into a normal HTML page. Small PHP sections are placed inside the HTML only where data needs to be checked or printed.

Line or section What it does
?> Ends the first PHP block. This lets the rest of the file start writing normal HTML.
<!DOCTYPE html> Tells the browser that this page is an HTML document.
<html> Starts the webpage structure.
<head> Starts the page information section. This part is not shown as page content.
<title>Cairns Weather</title> Sets the browser tab title.
<body> Starts the visible page content.
<h1>Cairns Current Weather</h1> Displays the main heading for the page.
<?php if (isset($data["current"])): ?> Checks whether the decoded API response has a current section before trying to display values from it.
<?php echo $data["current"]["temperature_2m"]; ?> Temporarily switches back into PHP and prints the temperature value inside the paragraph.
<p>...</p> Creates one paragraph for each weather value, such as temperature, humidity, rainfall and wind speed.
<?php else: ?> Provides an alternative message if the expected weather data was not available.
<?php endif; ?> Ends the PHP if statement that controls whether weather data or an error message is displayed.
</body> and </html> Close the visible page content and finish the HTML document.

Key pattern: PHP gets the data first, then HTML builds the page, and short PHP echo statements place selected API values into the HTML.

Starter example 2: Search for a place and display its weather

This file lets the user type a place name. It sends the place name to the geocoding API, uses the first matching result, then requests current weather for that location.

How the search page is built after the API setup

This example has two jobs: it shows a search form, then it displays weather data only after a place has been searched and a matching location has been found.

Line or section What it does
<h1>Search for Weather by Place</h1> Gives the page a clear heading so the user understands the purpose of the page.
<form method="get"> Creates a form that sends the place name through the URL as a GET value.
<label for="place"> Connects text to the input field so the form is clearer and more accessible.

Key pattern: the form collects a place, PHP uses that place to find coordinates, then the page only displays weather output if the API data exists.

How JSON becomes a PHP array

Open-Meteo sends JSON text. After json_decode($response, true), PHP can treat that JSON as a nested associative array. Each key leads to another section or value.

PHP · Nested key access
$data = json_decode($response, true); // JSON object → PHP array → nested key access $currentTemperature = $data["current"]["temperature_2m"]; $currentWindSpeed = $data["current"]["wind_speed_10m"];

Read this from left to right: start at $data, move into the current section, then select the temperature_2m value.

Code feature Why it matters
curl_init() Starts an API request.
CURLOPT_URL Sets the Open-Meteo endpoint.
CURLOPT_RETURNTRANSFER Stores the response so PHP can process it before output.
curl_exec() Sends the request and receives the response.
json_decode(..., true) Converts JSON into a PHP associative array.
isset(...) Checks that a section of the API response exists before trying to display it.

Common errors and what to check

What you might see What it usually means What to check first
Undefined index or Undefined array key Your code is trying to use a key that is not in the decoded JSON. Check the API URL includes the variable you are trying to display, then check the spelling of the nested keys.
null after decoding JSON The response may not be valid JSON, or the request may have failed. Open the API URL in a browser and check that JSON is returned.
No place appears after a location search The geocoding response may not contain a results array. Try a clearer place name, check spelling, and use isset($geoData["results"]) before reading the first result.
The graph or chart is blank The hourly arrays may be missing or shorter than expected. Check that the URL requests the hourly variable, such as hourly=temperature_2m.

Debugging habit

When something breaks, test in this order: API URL in browser → cURL response exists → JSON decodes → required keys exist → page output is built.

5Weather codes and icons

Open-Meteo returns a weather_code value. This is a number based on WMO weather interpretation codes. Your prototype can convert this number into a short condition label and a visual icon.

Important idea

Open-Meteo gives you the weather condition as data. It does not automatically draw the icon for you. Your program decides which label or icon matches each code.

Common weather code groups

Code or code group Meaning Possible icon idea
0 Clear sky Sun / clear day
1, 2, 3 Mainly clear, partly cloudy, overcast Partly cloudy / cloud
45, 48 Fog or depositing rime fog Fog
51, 53, 55 Drizzle Light rain
61, 63, 65 Rain Rain cloud
80, 81, 82 Rain showers Showers
95, 96, 99 Thunderstorm Thunderstorm

Starter PHP function: turn a code into a label

This example keeps the output simple. It helps you understand the logic before choosing your own icons or visual design.

PHP · Weather code to condition label
<?php function getWeatherLabel($code) {     if ($code == 0) {         return "Clear sky";     } elseif ($code == 1 || $code == 2 || $code == 3) {         return "Cloudy";     } elseif ($code == 45 || $code == 48) {         return "Fog";     } elseif ($code == 51 || $code == 53 || $code == 55) {         return "Drizzle";     } elseif ($code == 61 || $code == 63 || $code == 65) {         return "Rain";     } elseif ($code == 80 || $code == 81 || $code == 82) {         return "Rain showers";     } elseif ($code == 95 || $code == 96 || $code == 99) {         return "Thunderstorm";     } else {         return "Unknown condition";     } } $code = $data["current"]["weather_code"]; echo "<p>Condition: " . getWeatherLabel($code) . "</p>"; ?>

6Creating a simple graph from Open-Meteo data

Open-Meteo gives you the data for graphs. Your webpage needs to choose values from the JSON and turn them into HTML. A good way to learn this is to build a table first, change the same data into a simple visual bar graph, and then try a PHP-only SVG chart.

Important idea

The graph is not sent by Open-Meteo. Open-Meteo sends arrays of data. Your code pairs matching array values and decides how to display them.

Step 1: Understand the matching arrays

Hourly forecast data often uses matching arrays. The index number connects the time to the correct weather value.

Array position Time value Temperature value Meaning
[0] hourly.time[0] hourly.temperature_2m[0] The first time matches the first temperature.
[1] hourly.time[1] hourly.temperature_2m[1] The second time matches the second temperature.
[2] hourly.time[2] hourly.temperature_2m[2] The third time matches the third temperature.

Step 2: Display the paired data in a table first

This example assumes your earlier cURL code has already requested hourly=temperature_2m and decoded the JSON into $data.

PHP · Turn hourly arrays into a table
<h2>Hourly temperature table</h2> <?php $times = $data["hourly"]["time"]; $temps = $data["hourly"]["temperature_2m"]; echo "<table border='1'>"; echo "<tr><th>Time</th><th>Temperature</th></tr>"; for ($i = 0; $i < 8; $i++) {     $time = substr($times[$i], 11, 5);     $temp = $temps[$i];     echo "<tr>";     echo "<td>" . $time . "</td>";     echo "<td>" . $temp . "°C</td>";     echo "</tr>"; } echo "</table>"; ?>
Line-by-line explanation: table first
Line or idea What it does
$times = $data["hourly"]["time"]; Stores the hourly time array from the JSON.
$temps = $data["hourly"]["temperature_2m"]; Stores the matching hourly temperature array.
for ($i = 0; $i < 8; $i++) Loops through the first eight forecast hours.
substr($times[$i], 11, 5) Takes the time part from a value such as 2026-06-01T09:00.
$temps[$i] Gets the temperature that matches the same time position.
echo "<tr>"; Builds each table row using HTML output from PHP.

Step 3: Turn the same data into a simple graph

This example creates a basic bar-style graph. It uses the temperature value to decide the width of each bar.

PHP · Basic hourly temperature bar graph
<h2>Hourly temperature graph</h2> <?php $times = $data["hourly"]["time"]; $temps = $data["hourly"]["temperature_2m"]; for ($i = 0; $i < 8; $i++) {     $time = substr($times[$i], 11, 5);     $temp = $temps[$i];     $barWidth = $temp * 5;     echo "<p>";     echo $time . " — " . $temp . "°C ";     echo "<span style='display:inline-block; height:14px; width:" . $barWidth . "px; background:gray;'></span>";     echo "</p>"; } ?>
Line-by-line explanation: simple graph
Line or idea What it does
$barWidth = $temp * 5; Turns a weather value into a visual size. A temperature of 25 becomes a 125px bar.
echo $time . " — " . $temp . "°C "; Prints the label for each bar.
<span style='...'></span> Creates a simple horizontal bar inside the webpage.
width:" . $barWidth . "px; Uses the calculated width to make larger values appear as longer bars.

Trying a different weather variable

The same graph idea can be used with other hourly arrays, such as precipitation_probability or wind_speed_10m. The important part is to pair the time array with the matching weather array.

PHP chart option: using SVG

Because this assessment uses PHP, use PHP-generated output for charts. One option is to output an <svg> chart. SVG is still HTML-based page output, but PHP decides the labels, points and lines using the Open-Meteo JSON data.

Important idea

PHP collects and decodes the Open-Meteo JSON first. Then PHP loops through the hourly arrays and writes SVG shapes into the webpage. The chart is created from PHP output.

PHP · Simple SVG line chart for hourly temperature
<h2>Hourly temperature chart</h2> <?php $times = array_slice($data["hourly"]["time"], 0, 8); $temps = array_slice($data["hourly"]["temperature_2m"], 0, 8); $chartWidth = 520; $chartHeight = 220; $padding = 35; $minTemp = min($temps); $maxTemp = max($temps); $tempRange = $maxTemp - $minTemp; if ($tempRange == 0) {     $tempRange = 1; } $points = []; for ($i = 0; $i < count($temps); $i++) {     $x = $padding + ($i * (($chartWidth - 2 * $padding) / (count($temps) - 1)));     $y = $chartHeight - $padding - (($temps[$i] - $minTemp) / $tempRange * ($chartHeight - 2 * $padding));     $points[] = $x . "," . $y; } echo "<svg width='" . $chartWidth . "' height='" . $chartHeight . "' style='border:1px solid black;'>"; echo "<polyline points='" . implode(" ", $points) . "' fill='none' stroke='black' stroke-width='2' />"; for ($i = 0; $i < count($temps); $i++) {     $point = explode(",", $points[$i]);     $x = $point[0];     $y = $point[1];     $label = substr($times[$i], 11, 5);     echo "<circle cx='" . $x . "' cy='" . $y . "' r='4' fill='black' />";     echo "<text x='" . ($x - 12) . "' y='" . ($chartHeight - 10) . "' font-size='10'>" . $label . "</text>";     echo "<text x='" . ($x - 10) . "' y='" . ($y - 8) . "' font-size='10'>" . $temps[$i] . "°C</text>"; } echo "</svg>"; ?>
Line-by-line explanation: PHP SVG line chart
Line or idea What it does
array_slice(..., 0, 8) Selects the first eight hourly values so the chart stays readable.
$chartWidth, $chartHeight and $padding Set the size of the SVG drawing area and leave space for labels.
min($temps) and max($temps) Find the temperature range so values can be scaled to fit inside the chart.
$x = ... Spaces the points evenly from left to right.
$y = ... Converts the temperature into a vertical position on the chart.
<polyline> Draws the line that connects each temperature point.
<circle> Draws a dot for each hourly value.
<text> Adds time and temperature labels to the chart.

PHP-only chart option: rainfall chance columns

A column chart is useful when each time period is a separate value. This example uses precipitation_probability and draws each chance of rain value as an SVG rectangle.

PHP · Simple SVG column chart for chance of rain
<h2>Chance of rain chart</h2> <?php $times = array_slice($data["hourly"]["time"], 0, 8); $rainChance = array_slice($data["hourly"]["precipitation_probability"], 0, 8); $chartWidth = 520; $chartHeight = 220; $padding = 35; $barWidth = 35; $count = count($rainChance); $gap = ($chartWidth - 2 * $padding - ($count * $barWidth)) / ($count - 1); echo "<svg width='" . $chartWidth . "' height='" . $chartHeight . "' style='border:1px solid black;'>"; for ($i = 0; $i < $count; $i++) {     $chance = $rainChance[$i];     $barHeight = ($chance / 100) * ($chartHeight - 2 * $padding);     $x = $padding + ($i * ($barWidth + $gap));     $y = $chartHeight - $padding - $barHeight;     $label = substr($times[$i], 11, 5);     echo "<rect x='" . $x . "' y='" . $y . "' width='" . $barWidth . "' height='" . $barHeight . "' fill='gray' />";     echo "<text x='" . ($x + 3) . "' y='" . ($y - 5) . "' font-size='10'>" . $chance . "%</text>";     echo "<text x='" . ($x - 2) . "' y='" . ($chartHeight - 10) . "' font-size='10'>" . $label . "</text>"; } echo "</svg>"; ?>

Explaining graph choices in IA3 documentation

Do not only state that you made a graph. Explain why the graph type suits the weather data and user need.

  • Line graph: useful for temperature or wind trends because the order of time values matters.
  • Column graph: useful for rainfall chance or rainfall amount because it compares separate hourly or daily values.
  • Cards or icons: useful for quick current conditions because the user can scan them quickly.

Prompt: Explain why a line graph suits temperature change over time, while a column graph may suit rainfall chance or rainfall totals.

7Recommended cURL security and reliability options

The starter examples intentionally leave these choices out. Use this section as a list of ideas to consider and discuss in your IA3 documentation when explaining secure API request handling.

cURL option or function Purpose
CURLOPT_TIMEOUT Stops the request from hanging for too long.
CURLOPT_CONNECTTIMEOUT Limits how long the connection attempt can take.
CURLOPT_SSL_VERIFYPEER Verifies the HTTPS certificate.
CURLOPT_SSL_VERIFYHOST Checks that the certificate matches the host.
CURLOPT_FOLLOWLOCATION Controls whether redirects are followed.
CURLOPT_MAXREDIRS Limits the number of redirects.
CURLOPT_USERAGENT Identifies the application making the request.
CURLOPT_HTTPHEADER Can request JSON using a header such as Accept: application/json.
curl_getinfo() Can check the HTTP status code, such as 200, 404 or 500.
curl_error() Can help log or diagnose request errors.
Recommended secure API practices
  • Use HTTPS URLs only.
  • Set a timeout so the dashboard does not freeze if the API is slow.
  • Check the HTTP status code before trusting the response.
  • Decode the JSON and check that the required fields exist before displaying data.
  • Show a user-friendly message if the API call fails.
  • Do not show raw cURL, PHP or database errors to public users.
  • Log technical errors for administrators instead of displaying them publicly.

8Planning favourite locations

Your application should let logged-in users save favourite locations. Instead of copying a full table structure from this guide, use the JSON returned by Open-Meteo to decide which pieces of data your own method needs.

Think before storing

Forecast data changes regularly, so storing every temperature, rainfall or wind value may not be useful. A stronger approach is to store enough location information to request fresh forecast data when the dashboard loads.

Start with the geocoding JSON

When a user searches for a place, the geocoding API returns structured data. Look at the key-value pairs and decide which ones are needed for a saved favourite.

JSON · Example location result
{   "name": "Cairns",   "latitude": -16.923,   "longitude": 145.7663,   "country": "Australia",   "admin1": "Queensland",   "timezone": "Australia/Brisbane" }

Questions to guide your data choices

Question Why it matters
Which values are needed to call the forecast API again? The forecast API needs coordinates, so location data must support future weather requests.
Which values help the user recognise the saved place? A place name alone may not be enough if multiple towns or suburbs have the same name.
Which values help display the forecast in the correct local time? Time zone data can help avoid showing times in UTC or in the wrong region.
Which values link the favourite to the correct logged-in user? A favourite location should belong to the user who saved it, not every user of the system.
Which values are unnecessary or too sensitive to store? Good data design avoids storing information that is not needed for the purpose of the application.

Possible data pairs to consider

  • name and admin1 to help identify the place.
  • country and timezone to support clear display.
  • latitude and longitude to request forecast data.
  • A link between the logged-in user and the saved favourite.

Design decision to make

Decide which JSON values your prototype should store, which values it should only display, and which values it should request fresh from Open-Meteo each time.

Good prototype thinking

Use the geocoding JSON to create your own plan for favourite locations. Your final design should be justified by the purpose of the dashboard, the needs of the user, and the privacy/security expectations of the task.

Avoid

Do not copy every value from the API without a reason. Do not store large amounts of changing forecast data unless your design has a clear purpose for using historical records.
